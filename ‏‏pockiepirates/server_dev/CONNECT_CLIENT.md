# שלב חיבור הקליינט

מניחים שהשרת רץ אצלך כבר (ראית את `/status` בדפדפן). עכשיו מחברים את הקליינט.

המטרה של השלב הזה: לפתוח את `main_local.swf` ב־Ruffle, לראות את הקליינט מתחיל לדבר עם השרת, ולקבל לוגים שמראים איפה זה נשבר.

**אם זה לא יעבוד מיד — זה תקין.** הלוגים יספרו לי בדיוק מה חסר.

---

## שלב 1 — לעדכן את השרת לגרסה החדשה

הוספתי תמיכה ב־CORS (בקשות אבטחה שדפדפנים שולחים אוטומטית). אם השרת שלך רץ כרגע — סוגר אותו (Ctrl+C בחלון הפקודה שלו) ומעתיק שוב את הקובץ `server.py` המעודכן מהתיקייה כאן ל־`C:\server_dev\skeleton\` (להחליף את הקיים).

מפעיל מחדש:
```
cd /d C:\server_dev\skeleton
python server.py
```

ובודק שוב <http://127.0.0.1:8080/status> בדפדפן.

---

## שלב 2 — להפיק SWF מתוקן עם הכתובת המקומית

הקליינט המקורי מנסה להתחבר ל־`http://115.236.38.146:8080/` (השרת שמת). יש להחליף אותה ב־`http://127.0.0.1:8080/`. כתבתי לך סקריפט שעושה את זה אוטומטית.

בחלון פקודה **חדש** (השרת ממשיך לרוץ בחלון הקודם):

```
cd /d C:\server_dev\skeleton
python patch_swf.py "C:\נתיב\אל\pockiepirates\onepiece\main.swf"
```

תחליף את `C:\נתיב\אל\pockiepirates\onepiece\main.swf` בנתיב האמיתי לקובץ אצלך. הכי פשוט: גורר את הקובץ `main.swf` לתוך חלון הפקודה אחרי שהקלדת `python patch_swf.py ` (עם רווח), והנתיב יודבק אוטומטית.

הסקריפט יראה משהו כזה:

```
============================================================
  SWF patched successfully
  Replaced 1 occurrence of the original URL
    OLD: http://115.236.38.146:8080/
    NEW: http://127.0.0.1:8080/xxxx/
  Output: main_local.swf
============================================================
```

אם זה הצליח — נוצר קובץ חדש בשם `main_local.swf` באותה תיקייה ליד `main.swf` המקורי.

## שלב 3 — להפעיל ב־Ruffle

1. פותח את Ruffle (Standalone — האפליקציה הירוקה).
2. גורר אליו את הקובץ `main_local.swf` שנוצר. או File → Open SWF File.
3. מסתכל על המסך — יש כמה תרחישים:

### תרחיש א׳ — Ruffle טוען ומראה משהו

יופי. כל מה שתראה: מסך טעינה, מסך התחברות, או הודעת שגיאה. **לא חשוב מה.** עוברים לשלב 4.

### תרחיש ב׳ — Ruffle מראה "Unsupported AVM2 feature" או דומה

זה נפוץ עם Flex (המסגרת של המשחק). Ruffle מתקדם אבל עדיין לא תומך בהכל. במקרה הזה נצטרך לעבור ל־Flash Player Projector — תגיד לי ואני אתן מדריך.

### תרחיש ג׳ — Ruffle לא מצליח לטעון את הקובץ בכלל

ייתכן שה־patch שלי שיבר משהו. נבדוק את זה ביחד.

## שלב 4 — להסתכל על השרת

**ברגע שהקליינט נטען ב־Ruffle** — חוזר לחלון הפקודה של השרת. אמורות להופיע שורות חדשות, משהו כמו:

```
HTTP POST /xxxx/messagebroker/amf -> 200
  [!] UNHANDLED: loginService.someMethod
HTTP POST /xxxx/messagebroker/amf -> 200
  [!] UNHANDLED: roleService.getRoleInfo
```

**זה אדיר.** כל שורה כזאת היא בקשה שהקליינט שלח לשרת. עכשיו אנחנו יודעים מה הוא מבקש.

אם **לא רואים שורות כאלה** — הקליינט לא הצליח להתחבר. ראה "תרחישים" למטה.

## שלב 5 — לאסוף את הלוגים

עוצר את השרת (Ctrl+C בחלון הפקודה שלו).

הולך לתיקייה `C:\server_dev\logs\`. שני קבצים חדשים יהיו שם — אחד `transcript_*.jsonl` ואחד `unhandled_*.txt`.

**מצרף לי את שני הקבצים האלה בשיחה הבאה.** הם הכל מה שאני צריך כדי לכתוב את ה־handlers הבאים.

גם מועיל: צילום מסך של מה ש־Ruffle הראה.

---

## תרחישי שגיאה אפשריים

**Ruffle נפתח אבל המסך שחור / תקוע על Loading.**
זה אומר שהקליינט עדיין לא הצליח לדבר עם השרת. בודק:
1. השרת רץ? `/status` בדפדפן עובד?
2. ה־patch הצליח? יש קובץ `main_local.swf`?
3. פתחת את `main_local.swf` ולא את `main.swf`?

**Ruffle מראה Network Error או דומה.**
פתח את ה־Console של Ruffle (תפריט View → Toggle Web Inspector אם אופציה זמינה) — זה יראה איפה זה נשבר. צילום מסך של זה גם עוזר לי.

**אין שום שורה חדשה בלוג של השרת.**
הקליינט בכלל לא ניסה לדבר עם השרת. אולי ה־patch לא הצליח, אולי Ruffle חוסם רשת. נברר עם הצילומי מסך.

**Ruffle אומר "AMF Channel Error" / "Connection failed".**
דווקא טוב — הקליינט ניסה והשרת נפל. לוג השרת יראה למה.

## אם Ruffle בכלל לא עובד

אם Ruffle נפתח על SWF־ים פשוטים אבל קורס על main_local.swf, או מראה הודעת "Unsupported feature" — הסבירות גבוהה שהוא לא תומך בכל הפיצ'רים שהמשחק משתמש בהם. במקרה הזה עוברים ל־Flash Player Projector. 

תגיד לי שזה המצב ואני אתן מדריך נפרד להתקנת הפרויקטור (זה כולל הורדה מ־Internet Archive ועקיפה של ה־"Time Bomb" של אדובי).
